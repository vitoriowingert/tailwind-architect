# Tailwind Architect --- Sequential Development Roadmap

This document describes the **next steps in order** after the initial
monorepo, core engine, and CLI were generated by Cursor.

Current state (already implemented):

-   Monorepo workspace
-   Core engine modules
-   AST parser
-   CLI commands (analyze / fix / lint)
-   Tailwind config discovery
-   Project scanning with batching
-   Unit tests

Goal now: evolve the project into a **production‑ready dev tool**.

------------------------------------------------------------------------

# Phase 1 --- Validate the Core Engine

Goal: ensure the engine works correctly with real Tailwind code.

## Step 1 --- Run CLI on a real project

Example:

    tailwind-architect analyze ./src

Test against:

-   Next.js project
-   Tailwind project
-   shadcn/ui components

Recommended stack for testing:

-   Next.js
-   Tailwind CSS
-   shadcn/ui

------------------------------------------------------------------------

## Step 2 --- Add real‑world test cases

Create additional tests:

Examples:

    flex flex-row
    p-4 px-4
    bg-red-500 bg-blue-500
    pt-4 pb-4

Expected behavior:

-   detect redundancy
-   detect conflicts
-   suggest simplifications

------------------------------------------------------------------------

# Phase 2 --- Improve Tailwind Utility Resolution

Goal: increase conflict detection accuracy.

Current likely flow:

    string → tokens → rules

Future improved flow:

    string
     → variant parser
     → tailwind engine
     → css property mapping
     → conflict resolver

Tasks:

1.  Resolve utilities via Tailwind engine
2.  Map utility → CSS property
3.  Detect overlaps via property comparison

------------------------------------------------------------------------

# Phase 3 --- Variant Handling

Goal: fully support Tailwind variants.

Examples:

    hover:bg-red-500
    md:flex
    dark:bg-black
    group-hover:opacity-100

Implementation:

Parse variant stack:

    md:hover:bg-red-500

    variants = ["md","hover"]
    utility = "bg-red-500"

Conflicts must be evaluated **within the same variant scope**.

------------------------------------------------------------------------

# Phase 4 --- Framework Adapters

Current parsing supports JS/TS AST.

Add adapters for:

-   Vue SFC
-   Astro
-   Svelte

Architecture:

    adapters/
      react.ts
      vue.ts
      astro.ts
      svelte.ts

Each adapter extracts class strings from framework templates.
